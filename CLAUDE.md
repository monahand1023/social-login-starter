# AI-Assistant Context for social-login-starter

## Purpose

This repo helps a person — who may have no prior OAuth or AWS experience — add "Sign in with Google" and "Sign in with Apple" to a static website using AWS Cognito Hosted UI.

**When assisting a user with this repo: go one step at a time and ask for one value at a time.** Do not dump a full configuration at the user and ask them to fill it in. Work through `docs/01` → `docs/06` sequentially: confirm each prerequisite, gather each credential one by one, wait for the user's answer before continuing. This workflow prevents paste errors and CSRF/nonce mistakes that are hard to debug after the fact.

---

## The Mental Model

The sign-in flow has three actors:

1. **Your static site** (the files in this repo). It has no server — it's just HTML, CSS, and JS served from a local or hosted file server.
2. **AWS Cognito Hosted UI**. Cognito acts as the OAuth middleman. It talks to Google and Apple on your behalf, handles the sign-in page, validates the authorization code, and returns an ID token (a JWT) to your site.
3. **Google / Apple** (the Identity Providers). They verify the user's identity and tell Cognito "yes, this is alice@example.com." Your site never sees a Google or Apple password.

**The flow:**
```
User clicks "Sign in with Google"
  → js/auth.js generates a PKCE code verifier + challenge, builds the Cognito authorize URL
  → Browser redirects to Cognito Hosted UI
  → Cognito redirects to Google
  → User signs in at Google
  → Google redirects back to Cognito (/oauth2/idpresponse)
  → Cognito redirects back to your site (callback.html?code=...)
  → callback.html calls handleCallback() in js/auth.js
  → auth.js POSTs the code + code_verifier to Cognito (/oauth2/token)
  → Cognito returns an id_token + access_token
  → auth.js stores the tokens in sessionStorage and redirects to index.html
  → index.html reads the user's email/name from the id_token claims
```

**What `config.js` holds (4 values, all public):**
- `region` — AWS region (e.g. `us-east-1`)
- `userPoolId` — Cognito user pool ID (e.g. `us-east-1_AbC123xyz`)
- `clientId` — Cognito app client ID (long alphanumeric string, no secret)
- `domain` — Free Cognito Hosted UI domain (`<prefix>.auth.<region>.amazoncognito.com`)

These four values are public by design. The Cognito app client has no client secret, which is the correct configuration for browser-based apps. Committing them is safe and expected.

**What stays secret and never enters any committed file:**
- Google client secret
- Apple `.p8` private key

Both go into `infra/parameters.sh` (gitignored) and from there into AWS CloudFormation at deploy time. They never appear in `config.js` or any HTML/JS file.

## The #1 Gotchas — Enforce These

### 1. Never put a secret in `config.js` or any committed file
`config.js` is committed to the repo (after the user copies from `config.example.js`). It must contain only the 4 public Cognito values. The Google client secret and Apple private key go into `infra/parameters.sh` only — which is gitignored. If a user asks "where do I put my Google secret?", the answer is `infra/parameters.sh`, not `config.js`.

### 2. The Cognito domain is predictable — choose the prefix first
The Cognito Hosted UI domain follows this exact pattern:

```
<your-domain-prefix>.auth.<your-region>.amazoncognito.com
```

Both Google Cloud Console and Apple Developer require you to register this URL *before* you deploy. Because the formula is deterministic, the user can choose their `DOMAIN_PREFIX` (e.g. `my-app-login`) at the start of the Google setup doc (Step 2) and fill in the full redirect URL immediately — without waiting for the deployment to complete. Walk the user through choosing their prefix early, write it down, and use it consistently across both provider consoles and `parameters.sh`.

### 3. The app client must be public (no client secret)
In CloudFormation: `GenerateSecret: false`. In the AWS Console: choose "Public client" and do NOT tick "Generate client secret." A client secret cannot be kept secret in browser JS, and Cognito's token endpoint will reject calls that include an unexpected secret. If the user's client was created with a secret, they must recreate it as a public client — there is no way to remove the secret from an existing client.

### 4. Serve over `http://localhost:8000`, never `file://`
The callback URL registered with Google, Apple, and Cognito is `http://localhost:8000/callback.html`. Opening the HTML file directly via `file://` produces a different origin and a `redirect_uri_mismatch` error. Always instruct the user to run `npm run serve` (or `python3 -m http.server 8000`) and open `http://localhost:8000`.

### 5. `parameters.sh` and `*.p8` are gitignored — never commit them
`.gitignore` excludes `infra/parameters.sh` and all `*.p8` files. These contain real secrets. If a user accidentally stages them, instruct them to run `git rm --cached infra/parameters.sh` (or the `.p8` file) before committing. Revoking and regenerating credentials is the safe path if they were already pushed.

## How to Verify Changes

**Run the test suite:**
```bash
npm test
```
Runs `test/validate-config.test.js` (6 tests for the config validator) and `test/auth-helpers.test.js` (6 tests for PKCE, URL building, and JWT decode — includes the RFC 7636 test vector). No extra dependencies; uses Node's built-in `node:test`.

**Check JS files parse without error:**
```bash
node --check js/auth.js
node --check js/validate-config.js
node --check backend-optional/session-lambda/index.js
node --check backend-optional/session-lambda/cookie.js
```

**Validate the CloudFormation template (requires AWS CLI):**
```bash
aws cloudformation validate-template --template-body file://infra/cognito.yaml
```

**Check for syntax errors in deploy scripts:**
```bash
bash -n infra/deploy.sh
bash -n infra/parameters.example.sh
```

## Out of Scope

The following are intentionally not implemented. Do not add them without a clear request from the user:

- **LINE Login** — the architecture supports adding more Cognito IdPs, but LINE is not wired up and not documented.
- **Custom domains** — the repo uses the free `*.amazoncognito.com` Hosted UI domain. Custom domains require ACM certificates and additional CloudFormation resources.
- **Bilingual / internationalization** — the demo is English only.
- **MFA / TOTP** — Cognito supports MFA, but this starter does not configure or document it.
- **Refresh token rotation** — tokens are stored in sessionStorage and expire when the browser session ends. Persistent login and token refresh are left to the user to implement on top of this foundation.

## Architecture Source

This starter's auth pattern is modeled on a production e-commerce site. The core OAuth logic (`auth.js`) is a trimmed and generalized version of that site's auth layer. No real IDs, domains, secrets, or brand-specific details from the production site appear anywhere in this repo.
